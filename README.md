# ATLAS
### Advanced Telemetry Logging & Analytics System

ATLAS is a distributed data pipeline designed to ingest, process, and analyze high-frequency power telemetry data in near real-time. 

The system implements a **Lambda Architecture**, integrating both streaming and batch processing to transform raw telemetry JSON data into structured, deduplicated analytical datasets suitable for executive dashboards.

ATLAS uses technologies such as **Apache Kafka, Apache Spark, Delta Lake, ClickHouse, PostgreSQL, and Docker** to build a scalable and fault-tolerant telemetry analytics platform.

 

# System Architecture (Lambda)

The pipeline processes telemetry data generated by power monitoring systems through a dual-path architecture (Real-Time Stream + Hourly Batch).

```text
          Telemetry Generator (Microservice)
                   │
      ┌────────────┴────────────┐
      ▼                         ▼
 Kafka Broker (Stream)    Raw Directory (Batch)
      │      │                  │
      │      └───────┐          │
      ▼              ▼          ▼
ClickHouse Kafka   Spark Processing Engine
 Engine (Live)     (Explode & 1-Hour Window)
                     │
                     ▼
                 Delta Lake Storage
                 (ACID Deduplication MERGE)
                     │
                     ▼
                 ClickHouse & PostgreSQL
                     │
                     ▼
                 Analytics Dashboard

```

 

# Data Characteristics

Each telemetry JSON document simulates a massive overlap to test pipeline deduplication resilience:

* **7 days of historical telemetry readings** sent every hour.
* **2016 readings per file** per server.

Calculation:
`12 readings/hour × 24 hours × 7 days = 2016 readings`

### The Schema Challenge

Data arrives deeply nested. Readings are collected every **5 minutes** and stored inside a complex `PowerDetail` array.

Example Input Structure:

```json
{
  "server_name": "server-01",
  "device_id": "device-45",
  "PowerDetail": [
    { "Time": "10:00:00", "CpuWatts": 120, "AmbTemp": 24.5 },
    { "Time": "10:05:00", "CpuWatts": 118, "AmbTemp": 24.6 }
  ]
}

```

 

# Pipeline Components

## 1. Data Generation Layer (Jnana)

A Python FastAPI microservice generates telemetry JSON documents.

* Runs every **1 hour**.
* Generates a JSON payload following the `input_schema.py`.
* Includes **7 days of data history**, meaning 6 days and 23 hours of data are duplicates.

## 2. Streaming Layer (Nandini)

The payload is sent directly to an **Apache Kafka Cluster** (KRaft mode).

* Acts as the central streaming buffer.
* Enables sub-second latency for immediate threat/anomaly detection.

## 3. Raw Storage Layer (Jnana)

Simultaneously, the JSON payload is saved to physical volume storage (`/data/raw/`). This acts as our immutable batch fallback and data lake landing zone.

## 4. Data Processing Layer (Sanjula)

Apache Spark consumes the data (via Structured Streaming for Kafka, and Batch for the Raw folder).

* **Flattening:** Uses PySpark `explode()` to flatten the nested `PowerDetail` array into millions of individual rows.
* **Aggregation:** Applies a 1-hour tumbling window to roll up the 5-minute intervals.
* **Metrics:** Computes `avg_metric_value`, `max_metric_value`, and `min_metric_value`.

## 5. Delta Lake & Deduplication Layer (Manthan)

Acts as the gatekeeper for the Refined Layer.

* Uses **Delta Lake** `MERGE` (Upsert) operations to compare incoming flattened Parquet files against historical tables.
* Identifies the primary keys (`device_id` and `metric_time`) and mathematically strips the 6 days of duplicate data generated by the ingestion layer.
* Executes `OPTIMIZE` and `ZORDER` to compact small files and reduce query latency.

## 6. Analytics Layer (Varna)

Two analytical databases consume the clean data:

* **PostgreSQL:** Stores static metadata (server configurations, locations).
* **ClickHouse:** Utilizes the native Kafka Engine for raw live-streaming, and reads the perfectly deduplicated Delta Lake tables for accurate historical dashboards.
