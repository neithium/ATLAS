 
 
#   ATLAS : Advanced Telemetry Logging & Analytics System

ATLAS is a distributed data pipeline designed to ingest, process, and analyze high-frequency power telemetry data in near real-time. 

The system implements a **Lambda Architecture**, integrating both streaming and batch processing to transform raw telemetry JSON data into structured, deduplicated analytical datasets suitable for executive dashboards.

ATLAS uses technologies such as **Apache Kafka, Apache Spark, Delta Lake, ClickHouse, PostgreSQL, and Docker** to build a scalable and fault-tolerant telemetry analytics platform.

# Data Characteristics & Identity

### Device Identity Mapping

Each device is uniquely identified across the telemetry lake using a 3-way composite key:

* `platform_customer_id` (Minimum 10 characters)
* `application_customer_id` (Minimum 10 characters)
* `device_id`

### The Schema & Overlap Challenge

Data arrives deeply nested via an on-demand API. Readings are collected every **5 minutes** and stored inside a complex `PowerDetail` array.

To test pipeline deduplication resilience, the system handles a massive rolling overlap:

* **7 days of historical telemetry readings** sent every hour.
* **2016 readings per file** per server (`12 readings/hour × 24 hours × 7 days`).
* **Scale Target:** Engineered to handle a baseline of **100,000 devices**, heavily stress-tested for **400,000+ devices**.

Example Input Structure:

```json
{
  "platform_customer_id": "flipkart_prod_01",
  "application_customer_id": "acct_alpha_99",
  "device_id": "device-45",
  "PowerDetail": [
    { "Time": "10:00:00", "CpuWatts": 120, "AmbTemp": 24.5 },
    { "Time": "10:05:00", "CpuWatts": 118, "AmbTemp": 24.6 }
  ]
}

```
 

# Pipeline Components

## 1. Data Generation Layer (Jnana)

A Python FastAPI microservice generates telemetry JSON documents on demand.

* Reuses previously generated buffer data and appends only 1 hour of fresh data to reduce generation load.
* Outputs payloads matching the strict `input_schema.py`.

## 2. Streaming Layer (Nandini)

The payload is sent to an **Apache Kafka Cluster** operating in **KRaft mode** (Zookeeper-less).

* **Fault Tolerance:** 3 Brokers with a Replication Factor of 3.
* **Topic Distribution:** `raw-server-metrics` utilizes 12 partitions distributed evenly across leaders to maximize parallel throughput.
* **DLQ Pattern:** A dedicated `raw-server-metrics-dlq` (3 partitions) catches malformed packets, ensuring pipeline continuous uptime.

## 3. Data Processing Layer (Sanjula)

Apache Spark consumes the data, serving as the heavy computational engine.

* **Flattening & Transformation:** Uses PySpark `explode()` to un-nest arrays and enforces strict schema typing.
* **Aggregations:** Applies a 1-hour tumbling window (with watermarking) to roll up 5-minute intervals.
* **Resiliency:** Utilizes HDFS-backed checkpointing to store partition offsets. If a Spark node crashes, the system resumes processing exactly where it left off.

## 4. Delta Lake & Deduplication Layer (Manthan)

Acts as the **Source of Truth** and the gatekeeper for the Refined Storage Layer.

* **Storage Format:** Strictly utilizes **Parquet**. As a columnar format, it allows downstream analytics engines to read only required columns (e.g., `avg_metric_value`), bypassing irrelevant data and slashing disk I/O latency.
* **Deep Partitioning:** Implements a 5-level directory structure to enable extreme "Data Skipping" for 400k+ devices:
`/refined/metric_name/date/pcid/acid/device_id/`
* **ACID Deduplication:** Executes `MERGE` (Upsert) operations using the composite primary key (`device_id` + `metric_time` + `application_customer_id`). This mathematically strips the 6 days of duplicate data safely, backed by the Delta Transaction Log (`_delta_log`).
* **Performance Tuning:** Runs `OPTIMIZE` to compact small streaming files into 128MB Parquet blocks, neutralizing the "small file problem."

## 5. Analytics Layer (Varna)

Two analytical databases consume the clean data:

* **PostgreSQL:** Stores static metadata (server configurations, locations).
* **ClickHouse:** * *Fast Path:* Utilizes the native Kafka Engine for raw live-streaming severe alerts.
* *Batch Path:* A persistent Python scheduler pulls fully deduplicated Parquet data from the Refined Layer.
* *Validation:* ClickHouse runs a Daily (24h) aggregation Materialized View. This serves as a test case to ensure 100% mathematical parity with Spark's Hourly aggregations.
