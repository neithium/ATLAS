## 2. Problems Encountered & Solutions

### Problem 1: Syntax Error in `input_schema.py`

**Issue:** Missing comma in the schema definition on line 44.


**Learning:** Always validate schema files before running dependent pipelines.

### Problem 2: Column Name Mismatch (`metric_type` vs `report_type`)

**Error:**
```
AnalysisException: [UNRESOLVED_COLUMN.WITH_SUGGESTION] 
A column or function parameter with name `metric_type` cannot be resolved. 
Did you mean one of the following? [`metric_time`, `metric_id`, `MetricValue`, `report_type`, `max_metric_time`]
```

**Root Cause:** The `input_schema.py` defined `metric_type`, but the actual data generated by `generate_data.py` uses `report_type`.

**Solution:** Updated all references from `metric_type` to `report_type` in:
- `PipelineConfig.PARTITION_COLUMNS`
- `prepare_partition_columns()` function
- `verify_deduplication()` function
- Print statements

**Learning:** Always verify column names against the actual data schema, not just the input schema definition. The upstream data generator determines the real schema.

---

### Problem 3: Delta Table Schema Mismatch on Partition Change

**Error:**
```
AnalysisException: A schema mismatch detected when writing to the Delta table.
Partition columns do not match the partition columns of the table.
Given: [`report_type`, `partition_date`, `platform_customer_id`, `application_customer_id`, `device_id`]
Table: []
```

**Root Cause:** A previous run created a Delta table without partitioning. Subsequent runs with partitioning failed because Delta protects against accidental schema changes.

**Solution:** Added `overwriteSchema` option to the write operation:
```python
df.write
    .format("delta")
    .mode("overwrite")
    .option("overwriteSchema", "true")  # Allow partition schema changes
    .partitionBy(*partition_cols)
    .save(path)
```

**Learning:** When changing partition structure on Delta tables, explicitly set `overwriteSchema=true`. In production, consider using Delta table migration strategies instead of overwrite.

### Problem 4: Z-ORDER on Partition Column

**Error:**
```
IllegalArgumentException: [DELTA_ZORDERING_ON_PARTITION_COLUMN] 
device_id is a partition column. Z-Ordering can only be performed on data columns
```

**Root Cause:** Z-ORDER is a data clustering technique that reorganizes data within Parquet files. Partition columns are directory-level organization and not stored in the data files themselves.

**Original Design:**
- Z-ORDER BY `device_id` (for device-centric queries)

**Revised Design:**
- Z-ORDER BY `metric_time` (for time-series queries)

**Why `metric_time`?**
1. Most queries filter by time range first
2. Time-series data benefits from temporal clustering
3. `device_id` is already a partition column (automatic pruning)

**Learning:** Z-ORDER columns must be data columns, not partition columns. Choose Z-ORDER columns based on:
- Columns frequently used in WHERE clauses
- Columns NOT already in partition scheme
- High-cardinality columns that benefit from data locality

## 3. Final Architecture

### 3.1 Composite Primary Key (Triple-Hash)

```python
PRIMARY_KEY_COLUMNS = ["device_id", "metric_time", "application_customer_id"]
```

**MERGE Condition:**
```python
merge_condition = """
    target.device_id = source.device_id 
    AND target.metric_time = source.metric_time 
    AND target.application_customer_id = source.application_customer_id
"""
```

### 3.2 5-Level Partition Structure

```
/app/data/refined/
└── report_type=power_metrics/
    └── partition_date=2026-02-26/
        └── platform_customer_id=PLAT-12345/
            └── application_customer_id=APP-67890/
                └── device_id=SRV-101/
                    └── *.parquet
```

**Partition Column Order (matters for query pruning):**
1. `report_type` - Broadest filter (metric category)
2. `partition_date` - Date-based queries (extracted from `metric_time`)
3. `platform_customer_id` - Multi-tenant isolation
4. `application_customer_id` - Application-level isolation
5. `device_id` - Device-specific queries

### 3.3 Storage Configuration

| Setting | Value | Rationale |
|---------|-------|-----------|
| Format | Delta (Parquet internally) | ACID, time-travel, schema evolution |
| Compression | Snappy | Balance of speed and compression |
| Target File Size | 128 MB | Optimal for Spark parallelism |
| Z-ORDER Column | `metric_time` | Time-series query optimization |

## 6. Key Decisions Summary

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Column for metric type | `report_type` | Matches actual generated data schema |
| Date extraction | `partition_date` from `metric_time` | Enables date-based partition pruning |
| Z-ORDER column | `metric_time` | Can't use partition columns; time-series queries benefit most |
| Schema overwrite | Enabled | Needed for partition structure changes |
| MERGE strategy | Insert-only-if-new | Preserves existing records, strips 7-day overlap |
